1 Architecture Overview — The Big Picture

The system is designed as a 10-layer architecture, each layer independently scalable. Traffic flows top-to-bottom, with Kafka decoupling the write path (API) from the processing path (consumers).

2 Each Layer Explained
Layer 1 — Clients (1M+ Users)

Web browsers, mobile apps, IoT devices, and third-party webhooks. All communicate via HTTPS REST APIs.

Layer 2 — Edge / CDN (CloudFlare)
What It Does
DNS Resolution: Routes users to nearest data center
SSL Termination: Handles HTTPS encryption/decryption
DDoS Protection: Blocks malicious traffic before it reaches servers
Static Cache: Serves CSS, JS, images without hitting backend
Why It Matters at 1M Users
Without CDN, every static request hits your servers. With 1M users loading a page with 20 assets each, that's 20M requests/day just for static files. CDN absorbs 80%+ of that traffic.
Layer 3 — API Gateway
Routing
Routes /api/notifications/* to notification service, /api/users/* to user service. Single entry point.
Rate Limiting
Token bucket: 1000 req/min per user. Prevents abuse. Returns 429 Too Many Requests.
Authentication
JWT validation on every request. OAuth 2.0 flows. Role-based access control (RBAC).
Validation
Schema validation, input sanitization. Rejects malformed requests before they reach application layer.
Layer 4 — Load Balancer (Nginx / AWS ALB)

Distributes incoming requests across multiple Spring Boot instances using round-robin or least-connections. Performs health checks every 10 seconds — removes unhealthy instances automatically. Triggers auto-scaling when CPU > 70% or request queue > 100.

Layer 5 — Application Layer (Spring Boot Instances)

5-10 auto-scaled instances. Each instance runs: REST API endpoints, Kafka Producer, Async Thread Pool. Stateless design — any instance can handle any request. Session state stored in Redis (not in JVM).

Layer 6 — Kafka Cluster (3 Brokers, 30 Partitions)
Broker 1
KRaft mode (no Zookeeper). Stores partitions 0-9. Handles 1/3 of write traffic. SSD storage.
Broker 2
Stores partitions 10-19. Replicates data from Broker 1 for fault tolerance (replication factor = 3).
Broker 3
Stores partitions 20-29. If any broker dies, the other 2 have full copies — zero data loss.
Layer 7 — Consumer Layer (30 Parallel Consumers)

N consumer instances, each running 5 consumer threads. Total: 30 consumers across instances, one per partition. Each consumer: reads from Kafka → fetches from DB → resolves channel → sends notification → commits offset.

Layer 8 — Data Layer
PostgreSQL Primary
All write operations (INSERT, UPDATE).
WAL (Write-Ahead Log) streaming replication to replicas.
Connection pool: HikariCP (max 30 connections per instance).
Redis Cluster
User cache: Avoid DB lookup for every notification
Deduplication: Prevent duplicate sends after consumer restart
Rate limit state: Track per-user, per-channel send counts
ShedLock
Distributed scheduling lock stored in PostgreSQL.
Ensures @Scheduled tasks run on exactly 1 instance.
Prevents duplicate retry scans across N instances.

Layer 10 — Observability
Prometheus
Scrapes metrics from all instances. JVM stats, Kafka lag, HTTP latencies, custom business metrics.
Grafana
Visual dashboards. Alerts on: Kafka lag > 1000, error rate > 5%, P99 latency > 500ms.
ELK Stack
Centralized logs from all instances. Search, filter, and correlate logs. 30-day retention.
Jaeger
End-to-end distributed tracing. Trace a notification from HTTP POST → Kafka → Consumer → Email.

4 Scaling Strategy — How Each Layer Scales
SCALING DIMENSIONS HORIZONTAL SCALING (add more instances) ├── API instances: 5 → 10 → 20 (auto-scale on CPU/request rate) ├── Consumer instances: 6 → 12 → 30 (match Kafka partition count) ├── Kafka brokers: 3 → 5 → 7 (add when storage/throughput limited) ├── PG read replicas: 2 → 4 → 8 (add for more read throughput) └── Redis nodes: 3 → 6 (cluster mode, auto-sharding) VERTICAL SCALING (bigger machines) ├── PostgreSQL Primary: 4 vCPU → 8 → 16 (for write-heavy workloads) ├── Kafka brokers: 8GB → 16GB RAM (for larger page cache) └── Redis: 8GB → 32GB RAM (for larger cache) KAFKA-SPECIFIC SCALING ├── Add partitions: 3 → 30 → 100 (more parallelism) ├── Add consumer threads: match partition count ├── Add brokers: distribute partition load └── Increase retention: 7 days → 30 days (for replay capability) DATABASE SCALING ├── Connection pooling: HikariCP (30 per instance × 10 instances = 300 total) ├── Read/Write split: Writes → Primary, Reads → Replicas ├── Indexing: Composite indexes on (status, scheduledAt, priority) ├── Partitioning: Table partition by month for old notifications └── Archival: Move notifications older than 90 days to cold storage
Auto-scaling triggers (Kubernetes / ECS):
• Scale up API instances when: CPU > 70% OR request queue > 100 OR P99 > 500ms
• Scale up consumers when: Kafka consumer lag > 1000 messages for > 5 minutes
• Scale down when: CPU < 30% AND lag = 0 for > 15 minutes
5 Design Patterns Used
PatternWhereWhy
Publish-SubscribeKafka topicsDecouple producers from consumers; multiple consumer groups can read same topic
Event-Driven ArchitectureEntire systemServices communicate via events, not direct calls; async, resilient, scalable
Competing ConsumersConsumer groups30 consumers share work from 30 partitions; linear throughput scaling
Dead Letter QueueKafka DLT topicsPermanently failed messages don't block healthy ones; manual intervention queue
CQRSPostgreSQL replicasSeparate read and write paths; scale reads independently with replicas
Cache-AsideRedisCheck cache before DB; populate cache on miss; reduce DB load by 80%
Circuit BreakerExternal API callsResilience4j: If email provider fails, stop sending; retry after cooldown
Distributed LockShedLockEnsure scheduled tasks run on exactly 1 instance, not all N instances
Saga PatternNotification lifecyclePENDING → PROCESSING → SENT/FAILED → RETRY; each step is an event
Idempotent ConsumerKafka consumersCheck notification status before processing; safe to re-deliver messages
