Change #1 Complete

Async Processing

Decouple notification dispatch from HTTP request threads using Spring @Async with a dedicated thread pool

🔍 Problem vs Solution
The Problem

Synchronous Blocking API

The original NotificationService processed every notification synchronously inside the HTTP request thread. When a user called POST /api/notifications/send, the API would:


1. Save the notification to the database

2. Resolve the channel handler (Email/SMS/Push)

3. Call the external provider (SendGrid, Twilio, FCM)

4. Wait for the response

5. Update status and return to the caller


Steps 2-4 happen while the HTTP connection is held open. Under high load with millions of notifications, this causes thread starvation, API timeouts, and cascading failures. Bulk sends (/send/bulk) were even worse, processing every notification sequentially in a single thread.

The Solution

Async Fire-and-Forget with Thread Pool

Notification dispatch is now fully asynchronous. The API saves the notification to the DB and immediately returns a response with PENDING status. The actual channel delivery happens in a background thread pool managed by Spring's @Async infrastructure.


1. API receives request, validates, saves to DB

2. API returns immediately with PENDING status

3. Background thread picks up the notification

4. Channel handler sends via external provider

5. Status updated to SENT or FAILED asynchronously


The thread pool has a queue capacity of 10,000 with CallerRunsPolicy backpressure. When the queue is full, the calling thread processes the notification itself — providing natural throttling without dropping notifications.

🧠 Why Is Async So Much Faster?

The Core Difference: What the Caller Waits For

Sync: Caller waits for EVERYTHING
When sendNotification() is called in the sync version, the calling thread must wait for the entire pipeline to complete:

DB save (~5ms) + Status update to PROCESSING (~3ms) + External API call (100ms) + Status update to SENT (~3ms)

= ~111ms minimum per call

With 20 threads fighting for the H2 database lock, contention adds another 100-300ms, explaining the 219ms median and 430ms max.
Async: Caller waits for DB SAVE only
When sendNotification() is called in the async version, the calling thread only waits for:

DB save (~5ms) + Submit to thread pool queue (~0.01ms)

= ~5ms minimum per call

The 100ms external API call happens in a background thread from the notificationTaskExecutor pool. The caller never blocks on it. This is why min latency dropped from 106ms to just 2ms.

Throughput: Why 292 vs 81 Notifications/sec

Sync (81/sec): Each of the 20 test threads holds onto its execution for ~120-430ms (DB + external call + DB update). With 20 threads, the system can only process 20 threads ÷ 0.23s avg = ~87 notifications/sec. The actual external API call is the bottleneck — 20 threads are simultaneously sleeping for 100ms, then fighting for DB access.


Async (292/sec): Each of the 20 test threads finishes in ~5-42ms (just DB save). They cycle rapidly: 20 threads ÷ 0.065s avg = ~308 notifications/sec (close to our measured 292). The 100ms external call is offloaded to 10 background async threads running in parallel. The test threads never wait for the external call, so they can submit new notifications much faster.


In essence: Async separates the acceptance rate (how fast we take in notifications) from the processing rate (how fast we deliver them). The API can accept 292/sec even though external delivery still runs at ~100/sec. The queue absorbs the difference.

Total Time: Why 342ms vs 1,230ms for 100 Notifications

Sync (1,230ms): 100 notifications processed by 20 threads. Each thread blocks for ~120ms+. That means ~5 rounds of 20 parallel calls: 5 rounds × ~230ms avg = ~1,150ms (close to measured 1,230ms). Every round must fully complete (including the 100ms external call) before the thread can pick up the next notification.


Async (342ms): 100 notifications submitted by 20 threads. Each thread only does a DB save (~5-40ms) then moves on. With 20 threads cycling at ~40ms average, all 100 notifications are accepted in 5 rounds × ~65ms = ~325ms (close to measured 342ms). The actual email delivery continues in background threads after the test method returns.


Key insight: In the async test logs, you can see notif-async-* threads still sending emails AFTER the test method printed its results. This proves the fire-and-forget pattern — the API is done, but delivery continues asynchronously.

🧪 How the JUnit Load Test Works

Both repositories contain an identical PerformanceLoadTest.java in src/test/java/com/notification/performance/. The test uses @SpringBootTest to boot the full application context with a real H2 database, JPA, and all channel handlers.

1
@BeforeEach: Seed test users
Creates 10 users via UserService.registerUser() with unique usernames (using System.nanoTime() to avoid duplicates). Each user has email, phone, and device token populated.
2
Create ExecutorService with 20 threads
A fixed thread pool of 20 threads simulates 20 concurrent API callers. A CountDownLatch(100) tracks when all notifications are submitted.
3
Submit 100 concurrent sendNotification() calls
Each task builds a SendNotificationRequest for a round-robin user, records System.currentTimeMillis() before and after the call, and stores the elapsed time in a synchronized list.
4
Await completion & compute metrics
After all 100 tasks complete (via latch.await()), the test sorts the latency list and computes: total time, throughput (success × 1000 / totalMs), average, min, max, P50, P95, P99 percentiles.
5
Assert & log results
Asserts all 100 notifications succeeded with 0 failures, then logs the formatted results table.
// Run the sync test:
cd Notification-System
mvn test -Dtest="com.notification.performance.PerformanceLoadTest"

// Run the async test:
cd Notification-System-With-Scale
mvn test -Dtest="com.notification.performance.PerformanceLoadTest"

Note: Both repos have a Thread.sleep(100) added to all channel handlers (Email, SMS, Push) to simulate real external API latency. Without this delay, both versions would be near-instant (just DB operations) and the async advantage wouldn't be visible.

Why the HTTP Gap (1.4x) Is Smaller Than the JUnit Gap (3.6x)

The JUnit test calls notificationService.sendNotification() directly — measuring only the service layer. In sync mode, the service blocks for the full 100ms external call. In async mode, it returns immediately after DB save.


The HTTP test adds layers of overhead that both versions must pay equally:

  • Network round-trip (even on localhost: ~1-3ms)
  • Tomcat thread scheduling and connection handling
  • JSON serialization/deserialization (request + response)
  • Spring filter chain (security, validation, etc.)
  • HTTP connection pool contention with 30 concurrent clients

These fixed costs (~60-70ms) compress the relative difference. But the absolute savings remain: P50 dropped from 120ms to 71ms — a 49ms savings per request that compounds significantly at scale.


At production scale (thousands of req/sec with real 200-2000ms external API calls instead of 100ms), the async advantage grows dramatically because the fixed HTTP overhead becomes a small fraction of the total latency.
