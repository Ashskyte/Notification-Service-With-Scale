Change #1 Complete

Async Processing

Decouple notification dispatch from HTTP request threads using Spring @Async with a dedicated thread pool

🔍 Problem vs Solution
The Problem

Synchronous Blocking API

The original NotificationService processed every notification synchronously inside the HTTP request thread. When a user called POST /api/notifications/send, the API would:


1. Save the notification to the database

2. Resolve the channel handler (Email/SMS/Push)

3. Call the external provider (SendGrid, Twilio, FCM)

4. Wait for the response

5. Update status and return to the caller


Steps 2-4 happen while the HTTP connection is held open. Under high load with millions of notifications, this causes thread starvation, API timeouts, and cascading failures. Bulk sends (/send/bulk) were even worse, processing every notification sequentially in a single thread.

The Solution

Async Fire-and-Forget with Thread Pool

Notification dispatch is now fully asynchronous. The API saves the notification to the DB and immediately returns a response with PENDING status. The actual channel delivery happens in a background thread pool managed by Spring's @Async infrastructure.


1. API receives request, validates, saves to DB

2. API returns immediately with PENDING status

3. Background thread picks up the notification

4. Channel handler sends via external provider

5. Status updated to SENT or FAILED asynchronously


The thread pool has a queue capacity of 10,000 with CallerRunsPolicy backpressure. When the queue is full, the calling thread processes the notification itself — providing natural throttling without dropping notifications.

After (Asynchronous)

1
HTTP Request arrives at NotificationController
2
NotificationService.sendNotification() saves to DB
3
processNotificationAsync() submits to thread pool ASYNC
4
HTTP response returned immediately (PENDING) FAST

Background thread: send via channel, update status
API response time = DB save only (~5ms)
HTTP thread freed instantly
