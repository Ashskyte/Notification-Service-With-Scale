Kafka Fundamentals — Learning by Tracing a Real Notification

How Messages Flow Through Kafka

A complete walkthrough of what happens when you send a notification — from HTTP request to Kafka topic to consumer to email delivery. Every concept explained with our real code.

1 What Is Apache Kafka? — The Big Picture

Think of Kafka as a post office for your application. Instead of services calling each other directly (like a phone call), they drop messages into Kafka (like mailing a letter). The receiver picks up the message when they're ready.

THE POST OFFICE ANALOGY Traditional (Direct Call): NotificationService ──── phone call ────▶ ProcessingService │ "Hey, process notification #42 RIGHT NOW" │ "OK I'm busy, wait..." │ ← Tightly coupled "Still waiting..." │ Both must be alive "Done." │ Caller blocked │ Kafka (Message Queue): NotificationService ──── drop letter ────▶ 📬 Kafka Mailbox ────▶ ProcessingService (Topic) "Here's a letter about #42, "I'll hold it "I'll pick it up I'm done, bye!" safely on disk" when I'm ready" ✓ Sender returns instantly ✓ Message persisted ✓ Receiver works ✓ Doesn't care who reads it ✓ Survives crashes at own pace ✓ Fire-and-forget ✓ Multiple readers ✓ Can be on another can read same msg server entirely
Kafka Broker
The server that stores and serves messages. Think of it as the post office building. Runs on localhost:9092 in our setup (via Podman container).
Producer
Any code that sends messages to Kafka. In our app: NotificationKafkaProducer publishes events after saving to DB.
Consumer
Any code that reads messages from Kafka. In our app: NotificationKafkaConsumer listens on topics and processes notifications.
2 Topics — Named Mailboxes for Different Message Types

A Topic is a named channel where messages are stored. Think of it as a labeled mailbox. You create different topics for different types of messages. Messages in a topic are ordered, immutable, and persistent — once written, they stay there until the retention period expires.

OUR 4 KAFKA TOPICS ┌──────────────────────────────────────────────────────────────────────┐ │ KAFKA BROKER (localhost:9092) │ │ │ │ 📬 notification-send ← New notifications to process │ │ 3 partitions, high throughput │ │ │ │ 📬 notification-retry ← Failed notifications to retry │ │ 3 partitions, lower volume │ │ │ │ 📬 notification-send.DLT ← Permanently failed sends │ │ 1 partition, error queue (Dead Letter Topic) │ │ │ │ 📬 notification-retry.DLT ← Permanently failed retries │ │ 1 partition, error queue (Dead Letter Topic) │ │ │ └──────────────────────────────────────────────────────────────────────┘ Each topic is independent — messages in one topic don't affect another. Topics are created automatically by KafkaConfig.java when the app starts.
TopicPartitionsWho WritesWho ReadsWhen
notification-send 3 NotificationService (via KafkaProducer) KafkaConsumer → consumeSendEvent() Every new notification
notification-retry 3 RetryService (via KafkaProducer) KafkaConsumer → consumeRetryEvent() When a notification fails & needs retry
notification-send.DLT 1 Kafka error handler (automatic) KafkaConsumer → consumeSendDlt() After 3 failed processing attempts
notification-retry.DLT 1 Kafka error handler (automatic) KafkaConsumer → consumeRetryDlt() After 3 failed retry attempts
Key insight about Topics:
• A topic is NOT a queue — it's a log. Messages are appended at the end and never removed (until retention expires).
• Multiple consumers can read the same message (unlike a queue where the message disappears after reading).
• Each topic has a retention period (default 7 days). After that, old messages are cleaned up.
3 Partitions — Parallel Lanes Inside a Topic

A Partition is a sub-division of a topic. Think of a topic as a highway, and partitions as individual lanes. Messages are distributed across lanes, and each lane is processed independently — this is how Kafka achieves parallelism.

TOPIC = HIGHWAY, PARTITIONS = LANES Topic: notification-send (3 partitions) ┌─────────────────────────────────────────────────────────────────┐ │ │ │ Partition 0 ──▶ [msg] [msg] [msg] [msg] [msg] ──▶ Consumer C-0 │ │ offset: 0 1 2 3 4 │ │ │ │ Partition 1 ──▶ [msg] [msg] [msg] ──────────────▶ Consumer C-1 │ │ offset: 0 1 2 │ │ │ │ Partition 2 ──▶ [msg] [msg] [msg] [msg] ────────▶ Consumer C-2 │ │ offset: 0 1 2 3 │ │ │ └─────────────────────────────────────────────────────────────────┘ ✓ 3 partitions = 3 consumers can work in parallel = 3× throughput ✓ Each partition maintains its own ordering (offset 0, 1, 2, 3...) ✓ Messages within a partition are strictly ordered ✗ Messages across partitions have NO ordering guarantee
How Does a Message Know Which Partition to Go To?

When the producer sends a message, it provides a key. Kafka uses this formula:

Partition Assignment
// Kafka's partition assignment formula: partition = hash(key) % numberOfPartitions // In our application, key = notificationId (as String) // Example with 3 partitions: hash("1") % 3 = 0 → Partition 0 hash("2") % 3 = 2 → Partition 2 hash("3") % 3 = 1 → Partition 1 hash("4") % 3 = 0 → Partition 0 ← same partition as id=1 hash("5") % 3 = 2 → Partition 2
Why We Use notificationId as Key
Ordering: Same ID → same partition → messages for one notification are always in order
Idempotency: If we send+retry for the same notification, both go to the same partition
Even distribution: Different IDs spread evenly across partitions
What If Key Is null?
If no key is provided, Kafka uses round-robin — messages are distributed evenly but with no ordering guarantee. We always provide a key because ordering matters for notification processing.
Rule of thumb: Number of consumers in a group = number of partitions. We have 3 partitions and 3 consumer threads (concurrency=3 in KafkaConfig). Each consumer thread is assigned exactly 1 partition. If you add a 4th consumer, it will be idle because there are only 3 partitions to assign.
4 Offsets — The Bookmark That Tracks Where You Are

An Offset is a unique, sequential number assigned to each message within a partition. Think of it as a page number in a book. Each partition has its own set of offsets starting from 0.

OFFSETS — EACH PARTITION HAS ITS OWN COUNTER Partition 0: ┌────────┬────────┬────────┬────────┬────────┐ │ off: 0 │ off: 1 │ off: 2 │ off: 3 │ off: 4 │ ← newest message │ id:1 │ id:4 │ id:7 │ id:10 │ id:13 │ └────────┴────────┴────────┴────────┴────────┘ ▲ consumer has read up to here committed offset = 5 (next to read) Partition 1: ┌────────┬────────┬────────┐ │ off: 0 │ off: 1 │ off: 2 │ │ id:2 │ id:5 │ id:8 │ └────────┴────────┴────────┘ ▲ committed offset = 3 Key facts: ✓ Offset is per-partition, not per-topic (Partition 0 offset 3 ≠ Partition 1 offset 3) ✓ Offset only increases — never decreases, never resets ✓ Committed offset = "I've processed everything up to this point" ✓ If consumer crashes, it restarts from the last committed offset ✗ Offset does NOT correspond to notificationId — it's just a sequence number
What You See in Kafka UI
When you click Topics → notification-send → Messages, each message row shows:
Offset: 0, 1, 2... (position in that partition)
Partition: 0, 1, or 2 (which lane it's in)
Key: "1", "2", etc. (the notificationId)
Value: The JSON payload
Timestamp: When the producer sent it
Consumer Lag
Lag = Latest offset − Committed offset = messages waiting to be processed.
In Kafka UI: Consumers → notification-group shows lag per partition.
Lag = 0: Consumer is caught up (all messages processed)
Lag = 100: 100 messages waiting — consumer is falling behind
Lag growing: You need more consumers or faster processing
Why offsets matter for fault tolerance:
Our config uses enable.auto.commit = false and AckMode.RECORD. This means the offset is committed only after successful processing. If the consumer crashes mid-processing, it restarts from the last committed offset — the message is re-delivered and processed again. No data loss.
5 The Producer — How Messages Enter Kafka

Our producer is NotificationKafkaProducer.java. It takes a notification ID, wraps it in a lightweight NotificationEvent object, and sends it to a Kafka topic asynchronously.

PRODUCER INTERNALS — WHAT HAPPENS WHEN send() IS CALLED kafkaProducer.sendNotificationEvent(NotificationEvent.ofSend(42)) │ ▼ ┌──────────────────────────────────────────────────────────────┐ │ KafkaTemplate.send(topic, key, event) │ │ │ │ 1. Serialize the key │ │ StringSerializer: 42 → "42" (bytes) │ │ │ │ 2. Serialize the value │ │ JsonSerializer: NotificationEvent → JSON bytes │ │ {"notificationId":42,"eventType":"SEND", │ │ "timestamp":[2026,3,19,10,51,6]} │ │ │ │ 3. Determine partition │ │ hash("42") % 3 = Partition 0 │ │ │ │ 4. Add to producer buffer (batched for efficiency) │ │ Buffer fills up or linger.ms expires → flush to broker │ │ │ │ 5. Send to Kafka Broker │ │ Network request to localhost:9092 │ │ Broker writes to partition log file on disk │ │ │ │ 6. Wait for ACK (acks=all) │ │ All replicas confirm → broker sends ACK back │ │ │ │ 7. CompletableFuture completes │ │ .whenComplete() callback fires │ │ Logs: "Published [42] to [notification-send] p=0 o=5" │ └──────────────────────────────────────────────────────────────┘ The entire send() is non-blocking — returns a Future immediately The HTTP thread is NOT waiting for any of this to complete
NotificationKafkaProducer.java — The Key Lines
// 1. Create the event (just the ID, not the full notification) NotificationEvent event = NotificationEvent.ofSend(notification.getId()); // event = {notificationId: 42, eventType: "SEND", timestamp: now()} // 2. Use notification ID as the Kafka message key String key = String.valueOf(event.getNotificationId()); // key = "42" // 3. Send to Kafka — this returns immediately (non-blocking) CompletableFuture<...> future = kafkaTemplate.send(sendTopic, key, event); // 4. Callback fires when Kafka confirms receipt (or fails) future.whenComplete((result, ex) -> { if (ex != null) log.error("Failed: {}", ex.getMessage()); else log.info("Published [{}] partition={} offset={}", event.getNotificationId(), result.getRecordMetadata().partition(), result.getRecordMetadata().offset()); });
What does the JSON look like in Kafka?
The exact bytes stored in Kafka for notification ID 42:

Key: "42"
Value: {"notificationId":42,"eventType":"SEND","timestamp":[2026,3,19,10,51,6,937596200]}

This is what you see when you click a message in Kafka UI's Messages tab. The timestamp is serialized as an array [year, month, day, hour, minute, second, nano] by Jackson's default LocalDateTime serializer.
6 The Consumer — How Messages Leave Kafka

Our consumer is NotificationKafkaConsumer.java. It uses Spring's @KafkaListener annotation to automatically listen to topics. When a message arrives, the listener method is called.

Consumer Groups — Sharing the Work
CONSUMER GROUP: notification-group A Consumer Group is a team of consumers that work together to process messages from a topic. Kafka assigns partitions to consumers in the group. Rule: Each partition is assigned to exactly ONE consumer in the group. Topic: notification-send (3 partitions) Consumer Group: notification-group (3 consumer threads, concurrency=3) ┌─────────────────────────────────────────────────────────────┐ │ │ │ Partition 0 ─────────────────▶ Consumer Thread C-0 │ │ (processes id:1, 4, 7...) │ │ │ │ Partition 1 ─────────────────▶ Consumer Thread C-1 │ │ (processes id:2, 5, 8...) │ │ │ │ Partition 2 ─────────────────▶ Consumer Thread C-2 │ │ (processes id:3, 6, 9...) │ │ │ └─────────────────────────────────────────────────────────────┘ ✓ 3 partitions, 3 consumers = perfect balance (1:1 mapping) ✓ Each consumer processes messages independently in parallel ✗ If we had 4 consumers, 1 would be idle (no partition to read from) → If we had 2 consumers, 1 would handle 2 partitions (more work)

7 Live Case Study — Sending "Kafka UI Test" Email

Let's trace exactly what happened when we sent this test notification. Every step, every system, every log line.

The HTTP Request We Sent
Invoke-RestMethod ` -Uri "http://localhost:8080/api/notifications/send" ` -Method POST ` -ContentType "application/json" ` -Body '{"userId": 1, "title": "Kafka UI Test", "message": "This message should appear in Kafka UI topics!", "channel": "EMAIL"}'
What Happened — Step by Step
1
HTTP POST hits NotificationController
Spring MVC routes POST /api/notifications/send to NotificationController.sendNotification(). The request body is deserialized into SendNotificationRequest DTO.
2
NotificationService.sendNotification() called
Finds user with userId=1 from the database. Builds a Notification entity: title="Kafka UI Test", channel=EMAIL, priority=MEDIUM, status=PENDING.
3
Notification saved to database
notificationRepository.save(notification) → H2 assigns auto-generated id = 1. At this point the notification exists in DB with status = PENDING.
4
Kafka event published
kafkaProducer.sendNotificationEvent(NotificationEvent.ofSend(1))
Creates: {"notificationId": 1, "eventType": "SEND", "timestamp": [2026,3,19,...]}
Key = "1" → hash("1") % 3 → goes to Partition 0 (or whichever the hash maps to).
KafkaTemplate.send() returns a CompletableFuture — non-blocking.
5
HTTP 200 returned to client
The response is sent immediately — the HTTP thread doesn't wait for Kafka delivery or email sending. Client sees: {"status": "PENDING", "id": 1}
6
Kafka broker receives and stores the message
The message is written to the partition log file on disk. Broker sends ACK back to producer. Producer's whenComplete() callback logs:
Published [1] to [notification-send] partition=0, offset=0
7
Consumer Thread C-0 picks up the message
consumeSendEvent() is called with the deserialized NotificationEvent. The consumer was already polling the broker, waiting for new messages on Partition 0.
8
Consumer fetches full notification from DB
notificationRepository.findByIdWithUser(1) — loads the notification with User eagerly (JOIN FETCH to avoid lazy loading issue). Now we have the full entity with email address.
9
Email channel handler sends the email
processingService.processNotification(notification)
→ channelResolverService.resolve(EMAIL) → EmailChannelHandler
→ handler.send(notification) → logs: "EMAIL sent successfully for notification [1]"
→ status updated to SENT, sentAt = current timestamp.
10
Offset committed
consumeSendEvent() returned without exception → Spring Kafka commits offset. Consumer moves to the next message (if any). The committed offset is now 1 (meaning "I've processed everything up to offset 0, give me offset 1 next").

9 What Happens When Things Go Wrong
Scenario 1: Email Server Down — Consumer Fails
FAILURE FLOW — EMAIL SERVER UNREACHABLE Consumer C-0 picks up message {id:42, type:SEND} │ ├── Attempt 1: processNotification(42) │ └── EmailChannelHandler.send() → ConnectionRefusedException! │ └── DefaultErrorHandler catches exception │ └── Wait 1000ms... │ ├── Attempt 2: processNotification(42) (retry #1) │ └── EmailChannelHandler.send() → ConnectionRefusedException! │ └── Wait 1000ms... │ ├── Attempt 3: processNotification(42) (retry #2 — FINAL) │ └── EmailChannelHandler.send() → ConnectionRefusedException! │ └── All 3 attempts exhausted │ └── DeadLetterPublishingRecoverer kicks in: └── Publishes message to: notification-send.DLT └── Original offset committed (consumer moves on) └── DLT consumer logs: "Manual intervention required for id:42" What you see in Kafka UI: Topics → notification-send.DLT → Messages → the failed message is here
Scenario 2: Database Down — Notification Not Found
FAILURE FLOW — DATABASE UNREACHABLE Consumer C-0 picks up message {id:42, type:SEND} │ ├── Attempt 1: findByIdWithUser(42) │ └── JDBCConnectionException: Cannot acquire connection │ └── DefaultErrorHandler catches, wait 1s... │ ├── Attempt 2: findByIdWithUser(42) │ └── JDBCConnectionException (DB still down) │ └── Wait 1s... │ ├── Attempt 3: findByIdWithUser(42) │ └── JDBCConnectionException (still down) │ └── → notification-send.DLT → manual intervention When DB comes back online: Ops team can replay messages from DLT or manually process them
Scenario 3: Consumer Crashes Mid-Processing
CRASH RECOVERY — CONSUMER DIES AND RESTARTS Consumer C-0 picks up message {id:42} at offset 5 │ ├── findByIdWithUser(42) → OK ├── processingService.processNotification(42) ├── EmailChannelHandler.send() → email sent! ├── 💥 JVM CRASH before offset commit │ │ ... consumer restarts ... │ ├── Consumer reconnects to Kafka broker ├── Asks: "What's my last committed offset for Partition 0?" ├── Kafka says: "Offset 5" (the one we were processing) ├── Consumer re-reads message at offset 5 and processes it AGAIN │ └── This means the email might be sent twice! (This is called "at-least-once delivery") To handle this: Make your consumer IDEMPOTENT → Check notification status before processing → If already SENT, skip it
Kafka's delivery guarantees:
• At-most-once: Commit offset before processing → might lose messages (NOT recommended)
• At-least-once: Commit offset after processing → might duplicate messages (OUR approach)
• Exactly-once: Use Kafka transactions → complex, rarely needed for notifications

We use at-least-once because a duplicate email is better than a lost email. The consumer should check notification.getStatus() == SENT to skip duplicates.
