🏗 The Big Picture: 3 Pools Working Together

The async notification system has three distinct thread/connection pools that work together. Understanding how they interact is key to understanding performance and capacity.

📄 Step 1: Enable Async — NotificationSystemApplication.java

The very first thing needed is to tell Spring Boot: "allow @Async methods to run on separate threads." Without this, @Async annotations are silently ignored and everything runs synchronously.

NotificationSystemApplication.java
@SpringBootApplication @EnableScheduling @EnableAsync // ← THIS activates async proxy support public class NotificationSystemApplication { public static void main(String[] args) { SpringApplication.run(NotificationSystemApplication.class, args); } }
What @EnableAsync does internally: Spring creates a proxy around any bean that has @Async methods. When you call processingService.processNotificationAsync(notification), the proxy intercepts the call, wraps it in a Runnable, and submits it to the configured Executor — the caller returns immediately.
⚙ Step 2: Configure the Thread Pool — AsyncConfig.java

This is the core of the async architecture. This class creates a named ThreadPoolTaskExecutor bean with carefully chosen parameters. Every line matters.

AsyncConfig.java
@Slf4j @Configuration public class AsyncConfig { @Value("${notification.async.core-pool-size:10}") private int corePoolSize; // Always-alive threads @Value("${notification.async.max-pool-size:50}") private int maxPoolSize; // Burst capacity threads @Value("${notification.async.queue-capacity:10000}") private int queueCapacity; // Waiting room size @Bean(name = "notificationTaskExecutor") public Executor notificationTaskExecutor() { ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor(); executor.setCorePoolSize(corePoolSize); // 10 threads always alive executor.setMaxPoolSize(maxPoolSize); // Can grow to 50 under load executor.setQueueCapacity(queueCapacity); // 10,000 tasks can wait executor.setThreadNamePrefix("notif-async-"); // Easy log identification // CRITICAL: What happens when queue is full AND all 50 threads are busy executor.setRejectedExecutionHandler( new ThreadPoolExecutor.CallerRunsPolicy() ); // CRITICAL: On app shutdown, don't kill in-flight notifications executor.setWaitForTasksToCompleteOnShutdown(true); executor.setAwaitTerminationSeconds(60); executor.initialize(); return executor; } }
🧩 Every Parameter Explained: Why These Numbers?
corePoolSize 10
The number of threads that are always alive, even when idle. These threads are pre-created and waiting for work. They never get destroyed (unless allowCoreThreadTimeOut is set to true).
Why 10? This matches the default HikariCP connection pool size (10). Since each async thread needs a DB connection to update notification status, having more core threads than DB connections would just cause threads to wait for connections. 10 core threads = 10 DB connections = perfect alignment for steady-state load.

Rule of thumb: corePoolSize ≤ DB connection pool size for I/O-bound work. For CPU-bound work, use Runtime.getRuntime().availableProcessors().
maxPoolSize 50
The maximum number of threads the pool can grow to when the queue is full. These extra threads (11-50) are only created when the queueCapacity (10,000) is exhausted. They are temporary — destroyed after being idle for 60 seconds (default keepAlive).
Why 50? This is the burst safety net. Under extreme traffic (10,000+ queued notifications), the pool scales up to 50 threads to drain the queue faster. 50 is chosen because:

1. Memory: Each thread uses ~1MB of stack space. 50 threads = ~50MB — acceptable.
2. Context switching: Beyond ~50-100 threads, the OS spends more time switching between threads than doing actual work. 50 keeps contention manageable.
3. External API rate limits: If the email provider allows max 50 concurrent connections, having 50 threads naturally aligns with that limit.
4. DB connections: Even though HikariCP defaults to 10, these burst threads will spend most of their time in Thread.sleep(100) (the external API call), only briefly needing a DB connection.
queueCapacity 10,000
The size of the LinkedBlockingQueue that holds tasks when all core threads are busy. Tasks wait here until a thread becomes available to process them. This is the "waiting room."
Why 10,000? This allows the system to absorb massive traffic spikes without rejecting requests.

Math: If each notification takes ~100ms to process, and we have 10 core threads, the pool processes 10 × (1000/100) = 100 notifications/sec. A queue of 10,000 means we can absorb a 100-second burst of traffic at any rate without rejection.

Memory cost: Each queued task is a lightweight Runnable referencing a Notification object (~200 bytes). 10,000 × 200B ≈ 2MB — negligible.

Tradeoff: Too large → notifications wait too long in queue (stale by the time they're sent). Too small → system starts rejecting tasks under moderate spikes. 10,000 is a safe middle ground.
threadNamePrefix "notif-async-"
Every thread created by this pool will be named notif-async-1, notif-async-2, etc. This makes it trivial to identify async notification threads in logs, thread dumps, and monitoring tools.
Why it matters: When debugging production issues, you can run jstack <PID> and instantly see which threads are processing notifications vs. Tomcat HTTP threads vs. scheduler threads. Without this, all threads would be named pool-N-thread-M — impossible to diagnose.
rejectedExecutionHandler CallerRunsPolicy
This defines what happens when a new task arrives but all 50 threads are busy AND the 10,000 queue is full. With CallerRunsPolicy, the task runs on the caller's thread (the Tomcat HTTP thread) instead of being rejected.
Why CallerRunsPolicy and not others?

Policy Behavior Risk
AbortPolicy (default) Throws RejectedExecutionException Notification is LOST. User sees HTTP 500.
DiscardPolicy Silently drops the task Notification is LOST. No error, no log. Worst option.
DiscardOldestPolicy Drops the oldest queued task Oldest notification is LOST. Unpredictable.
CallerRunsPolicy ✓ Caller thread runs the task itself No notification lost. Caller just slows down (natural backpressure).

The beauty of CallerRunsPolicy: It creates natural backpressure. When the system is overwhelmed, Tomcat threads slow down because they're doing the work themselves. This automatically reduces the rate of incoming requests — the system self-regulates without losing any data.
waitForTasksToCompleteOnShutdown true
When the application shuts down (deploy, restart, crash recovery), this ensures the executor does not kill in-flight notification tasks. It waits up to awaitTerminationSeconds (60s) for all currently executing and queued tasks to finish.
Without this: If 500 notifications are in-flight and you restart the server, all 500 are lost — they're in PROCESSING status in DB but never complete.

With this: The shutdown signal tells the pool to stop accepting new tasks but finish existing ones. If tasks don't finish within 60 seconds, they're interrupted (and the retry service will pick them up later).
🔄 Thread Pool Execution Order (Critical to Understand)

Java's ThreadPoolTaskExecutor does NOT immediately create threads up to maxPoolSize. It follows a specific order that many developers get wrong:

New task arrives │ ▼ ┌─────────────────────────────────┐ │ Are there idle core threads? │ │ (fewer than corePoolSize=10 │ │ threads exist) │ └────────┬──────────┬─────────────┘ YES │ │ NO │ │ ▼ ▼ ┌──────────┐ ┌─────────────────────────┐ │ Create a │ │ Is the queue full? │ │ new core │ │ (queue size ≥ 10,000) │ │ thread │ └────────┬──────────┬──────┘ │ to run │ NO │ │ YES │ the task │ │ │ └──────────┘ ▼ ▼ ┌──────────┐ ┌──────────────────────────┐ │ Add task │ │ Are there fewer than │ │ to the │ │ maxPoolSize=50 threads? │ │ queue │ └────────┬──────────┬───────┘ │ (wait) │ YES │ │ NO └──────────┘ │ │ ▼ ▼ ┌──────────┐ ┌──────────────────┐ │ Create a │ │ REJECT! │ │ temporary│ │ CallerRunsPolicy │ │ thread │ │ → caller thread │ │ (11-50) │ │ runs the task │ └──────────┘ └──────────────────┘
Key insight: The pool fills the queue FIRST before creating extra threads. This means threads 11-50 only get created when the queue hits 10,000 — which almost never happens under normal load. In practice, the pool runs with 10 core threads and the queue absorbs all bursts.

⚡ Step 4: The @Async Method — NotificationProcessingService.java

This is where the actual thread handoff happens. The @Async("notificationTaskExecutor") annotation tells Spring to run this method on the notificationTaskExecutor pool instead of the caller's thread.

NotificationProcessingService.java
@Async("notificationTaskExecutor") // ← Run on our custom thread pool @Transactional public CompletableFuture<Void> processNotificationAsync(Notification notification) { processNotification(notification); // Heavy work here return CompletableFuture.completedFuture(null); // Already done } @Transactional public void processNotification(Notification notification) { try { notification.setStatus(NotificationStatus.PROCESSING); notificationRepository.save(notification); // DB write #1 NotificationChannelHandler handler = channelResolverService.resolve(notification.getChannel()); handler.send(notification); // 100ms external API call notification.setStatus(NotificationStatus.SENT); notification.setSentAt(LocalDateTime.now()); notificationRepository.save(notification); // DB write #2 } catch (Exception e) { notification.setStatus(NotificationStatus.FAILED); notification.setFailureReason(e.getMessage()); notificationRepository.save(notification); // DB write (failure) retryService.retryNotification(notification); // Auto-retry } }
Why CompletableFuture<Void> and not void? Returning CompletableFuture allows the caller to optionally .thenAccept() or .exceptionally() the result if needed. In our case, we use fire-and-forget, but the return type gives us flexibility for future enhancements (e.g., waiting for batch completion).
📨 Step 5: The Caller — NotificationService.java

This is where the Tomcat HTTP thread hands off work and returns immediately.

NotificationService.java (sendNotification method)
@Transactional public NotificationResponse sendNotification(SendNotificationRequest request) { User user = userService.findUserById(request.getUserId()); // DB read Notification notification = Notification.builder() .user(user) .title(request.getTitle()) .message(request.getMessage()) .channel(request.getChannel()) .priority(priority) .status(NotificationStatus.PENDING) .build(); notification = notificationRepository.save(notification); // DB write // ↓↓↓ THIS IS THE KEY LINE ↓↓↓ // Spring proxy intercepts this call, wraps it in a Runnable, // submits to notificationTaskExecutor, and returns IMMEDIATELY processingService.processNotificationAsync(notification); // This line executes in ~0.01ms, NOT after email is sent return NotificationResponse.fromEntity(notification); }

🚀 How It Handles High Traffic — Capacity Math

Let's calculate exact capacity for different traffic levels:

Traffic Level Incoming Rate Active Threads Queue Usage Processing Rate Status
Low 10 req/sec 1-2 of 10 core 0 (empty) 100/sec capacity ✅ Healthy
Normal 50 req/sec 5-6 of 10 core 0 (empty) 100/sec capacity ✅ Healthy
High 100 req/sec 10 of 10 core ~0 (slight buildup) 100/sec capacity ⚠ At capacity
Spike 500 req/sec 10 core (busy) ~400/sec accumulating 100/sec draining ⚠ Queue absorbing
Extreme Spike
(25 seconds)		 500 req/sec × 25s 10 core (busy) ~10,000 (FULL) 100/sec draining 🔴 Queue full → burst threads
Beyond Extreme 500 req/sec 50 max (all busy) 10,000 (FULL) 500/sec capacity ⚠ CallerRunsPolicy kicks in
Processing rate formula:
// Each notification takes ~100ms (simulated external API call) // So each thread can process 1000ms / 100ms = 10 notifications/sec Processing Rate = activeThreads × (1000 / avgProcessingTimeMs) // With 10 core threads: 10 × 10 = 100 notifications/sec // With 50 max threads: 50 × 10 = 500 notifications/sec // With real APIs (500ms): 10 × 2 = 20 notifications/sec (core) // 50 × 2 = 100 notifications/sec (burst)
Total system capacity before any data loss:
Queue (10,000 tasks) + burst processing (50 threads × 10/sec = 500/sec) + CallerRunsPolicy (backpressure, never drops).
Result: Zero notifications are ever lost, regardless of traffic level.
🗄 HikariCP: The Database Connection Pool

Every DB operation (save notification, update status, lookup user) requires a database connection. Spring Boot auto-configures HikariCP as the connection pool — the fastest Java connection pool.

application.properties (current defaults)
# H2 Database (current — will be replaced with PostgreSQL) spring.datasource.url=jdbc:h2:mem:notificationdb spring.datasource.driverClassName=org.h2.Driver # HikariCP defaults (not explicitly set — Spring Boot defaults): # spring.datasource.hikari.maximum-pool-size = 10 # spring.datasource.hikari.minimum-idle = 10 (same as max) # spring.datasource.hikari.idle-timeout = 600000 (10 min) # spring.datasource.hikari.connection-timeout = 30000 (30 sec) # spring.datasource.hikari.max-lifetime = 1800000 (30 min)

Important: Async threads only hold a DB connection briefly (~3-5ms for each save). Between DB operations, the thread is doing the external API call (100ms) and does NOT hold a connection. Spring's @Transactional releases the connection back to the pool after each transactional method. This means 10 DB connections can serve 50 async threads efficiently.
🎯 Thread Count Selection: How to Choose the Right Numbers

Thread pool sizing depends on the type of work your threads are doing. Here's the decision framework:

CPU-Bound Work
Threads doing computation (encryption, compression, JSON processing).

Formula:
threads = CPU cores or CPU cores + 1

Why: More threads than cores = context switching waste. Each thread runs at 100% CPU; adding more just slows everyone down.

Example: 8-core machine → 8-9 threads max.
I/O-Bound Work ← Our Case
Threads waiting on external calls (HTTP APIs, DB, file I/O).

Formula:
threads = CPU cores × (1 + wait_time / compute_time)

For our case:
CPU cores = 8, wait = 100ms (email API), compute = ~5ms (DB).
8 × (1 + 100/5) = 8 × 21 = 168 threads

But we use 10-50. Why? Because DB connection pool (10) is the bottleneck, not CPU. More threads would just wait for DB connections.

Our Specific Choices Explained

Parameter Value Reasoning
corePoolSize 10 = HikariCP max connections. Each core thread can always get a DB connection without waiting.
maxPoolSize 50 5× core for burst handling. Burst threads spend 95% of time in external API call (no DB needed). Only briefly need a DB connection for status update.
queueCapacity 10,000 = 100 seconds of traffic at processing capacity (100/sec). Absorbs any realistic spike.
HikariCP pool 10 (default) H2 in-memory DB is fast (~1ms per query). 10 connections handle thousands of requests/sec. For PostgreSQL, increase to 20-30.
Tomcat threads 200 (default) Each HTTP request uses a Tomcat thread for ~8ms (async), so 200 threads can handle 200 × (1000/8) = 25,000 req/sec. Not the bottleneck.
📈 Production Scaling Guide

When moving to production with real email APIs (200-2000ms latency) and PostgreSQL, adjust these values:

Parameter Current (Dev) Production (Recommended) Why
corePoolSize 10 25 Match increased DB pool. Real API calls take 500ms+, need more concurrent threads.
maxPoolSize 50 100 Real external APIs are 5-20× slower. Need more burst threads to maintain throughput.
queueCapacity 10,000 50,000 Longer processing time per notification = slower drain rate = bigger queue needed.
HikariCP pool 10 25-30 PostgreSQL can handle 100+ connections. Match core thread count.
Tomcat threads 200 200 (keep default) Async makes Tomcat threads fast (~8ms each). 200 is more than enough.
Warning: Anti-pattern to avoid
Don't set corePoolSize = 200 thinking "more threads = more speed". This causes:
• 200 threads competing for 10 DB connections = 190 threads always waiting
• Massive context switching overhead
• Memory waste (200 × 1MB = 200MB of stack space)
• Rule: threads should never exceed your slowest shared resource (DB connections) by more than 5-10×.
🛡 Error Handling in Async Mode

Since the HTTP response is already sent by the time the email is processed, failures must be handled differently:

1
Exception in channel handler
If handler.send(notification) throws, the catch block sets status to FAILED with the failure reason and saves to DB. The notification is never lost.
2
Automatic retry via RetryService
The catch block calls retryService.retryNotification(notification) which uses exponential backoff (1s → 2s → 4s) up to maxRetries=3.
3
Permanent failure tracking
After 3 failed retries, the notification is marked FAILED permanently with the failureReason recorded in the database. Ops team can query failed notifications.
4
Scheduled retry sweep
A @Scheduled job runs periodically, finds all FAILED notifications that haven't exceeded max retries, and processes them again. This catches edge cases like thread pool shutdown during processing.
