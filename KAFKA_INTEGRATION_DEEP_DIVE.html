Apache Kafka Integration

Decoupling the Notification System with persistent message queues — replacing in-memory @Async with fault-tolerant, horizontally scalable event-driven processing.

1 The Problem — Why @Async Is Not Enough

Our previous architecture used Spring's @Async with a ThreadPoolTaskExecutor to process notifications in background threads. While this was a huge improvement over synchronous processing, it has fundamental limitations when operating at scale.

CURRENT ARCHITECTURE — @Async with In-Memory Thread Pool ┌─────────────────┐ ┌─────────────────────────────────────┐ │ HTTP Client │ │ Spring Boot Application │ │ │ │ │ │ POST /api/send │ ───────▶│ NotificationService │ │ │ │ │ │ └─────────────────┘ │ ▼ │ │ notificationRepository.save() │ │ │ │ │ ▼ │ │ processingService │ │ .processNotificationAsync() │ │ │ │ │ ▼ │ │ ┌───────────────────────────────┐ │ │ │ ThreadPoolTaskExecutor │ │ │ │ ┌───────────────────────┐ │ │ │ │ │ In-Memory Queue │ │ │ │ │ │ (capacity: 10,000) │ │ │ │ │ └───────┬───────────────┘ │ │ │ │ │ │ │ │ │ ┌───────┼───────┐ │ │ │ │ │ T-1 │ T-2 │ ...T-50 │ │ │ │ └───┬───┴───┬───┘ │ │ │ └───────┼───────┼───────────────┘ │ │ ▼ ▼ │ │ Email/SMS/Push handlers │ └─────────────────────────────────────┘ █ Everything runs inside ONE JVM — one single point of failure
⚠ Critical Risks
1. Data Loss on Crash — All 10,000 queued tasks vanish if JVM dies
2. No Replay — Failed events cannot be re-consumed or audited
3. Single Instance — Cannot distribute work across multiple servers
4. Tight Coupling — Producer and consumer share the same JVM process
5. No Backpressure — When queue is full, CallerRunsPolicy blocks HTTP threads
6. No Ordering — Thread pool does not guarantee message ordering
✅ What Kafka Solves
1. Persistent Queue — Messages survive broker and app restarts
2. Replay Capability — Re-read messages from any offset at any time
3. Horizontal Scaling — Add consumer instances, Kafka auto-rebalances
4. Full Decoupling — Producer and consumer are independent processes
5. Backpressure Built-in — Consumers pull at their own pace
6. Partition Ordering — Same key always goes to same partition in order
Real-world disaster scenario: Imagine 50,000 notifications are queued in the @Async thread pool during a Black Friday sale. The application crashes due to an OutOfMemoryError. All 50,000 notifications are permanently lost — users never get their order confirmations. With Kafka, every single message is persisted to disk before acknowledgment — zero data loss even during crashes.

Key insight: The HTTP thread no longer waits for notification processing. It saves to DB, publishes a lightweight event (just the notification ID) to Kafka, and returns immediately. The consumer — running in a separate thread (or even a separate server) — picks up the event and does the heavy work.
3 Kafka Topics — Our Message Channels

We define 4 Kafka topics — 2 main processing topics and 2 dead letter topics for failure handling.

notification-send 3 partitions
Purpose: Main topic for new notifications that need processing and delivery.
Message Key: notificationId — same notification always goes to the same partition.
Producer: NotificationService publishes after saving notification to DB.
Consumer: NotificationKafkaConsumer.consumeSendEvent() fetches from DB and calls processNotification().
Volume: High — every new notification passes through this topic.
notification-retry 3 partitions
Purpose: Topic for notifications that failed and need to be retried via Kafka.
Message Key: notificationId
Producer: NotificationKafkaProducer.sendRetryEvent() publishes retry events.
Consumer: NotificationKafkaConsumer.consumeRetryEvent() calls retryService.executeRetry().
Volume: Low — only failed notifications that need event-driven retries.
notification-send.DLT Dead Letter
Purpose: Messages from notification-send that failed all 3 processing attempts.
Partitions: 1 (low volume — this is an error queue).
Consumer: consumeSendDlt() logs the error for manual investigation.
Action: Operations team reviews, fixes root cause, and replays.
notification-retry.DLT Dead Letter
Purpose: Messages from notification-retry that failed all processing attempts.
Partitions: 1 (low volume).
Consumer: consumeRetryDlt() logs permanently failed retries.
Action: Permanent failures requiring human intervention.

4 Message Flow — Step by Step
Flow 1: Sending a New Notification (Happy Path)
1
HTTP Request arrives at POST /api/notifications/send
Controller validates the request and calls NotificationService.sendNotification().
2
Notification saved to database with status = PENDING
notificationRepository.save(notification) — the notification gets a DB-generated primary key ID.
3
Event published to Kafka topic notification-send
kafkaProducer.sendNotificationEvent(NotificationEvent.ofSend(id))
The message key = notificationId — this ensures ordering per notification.
4
HTTP 200 returned immediately to the client
Client gets NotificationResponse with status=PENDING. The actual sending happens asynchronously via Kafka.
5
Kafka Consumer picks up the event
consumeSendEvent() receives the event, fetches the full Notification entity from DB using the ID.
6
Notification processed and delivered
processingService.processNotification() resolves the channel handler (Email/SMS/Push) and sends. Status updated to SENT on success, or FAILED → scheduleRetry() on failure.

5 The Event DTO — NotificationEvent.java

The Kafka message carries minimal data: just the notification ID, event type, and timestamp. The full notification data stays in the database — the consumer fetches it by ID. This is the event-carried state transfer pattern.

com/notification/event/NotificationEvent.java
package com.notification.event; import lombok.*; import java.io.Serializable; import java.time.LocalDateTime; @Getter @Setter @NoArgsConstructor @AllArgsConstructor @Builder @ToString public class NotificationEvent implements Serializable { private Long notificationId; // DB primary key — consumer fetches full entity private String eventType; // "SEND" or "RETRY" private LocalDateTime timestamp; // When the event was created // Factory method: create a SEND event public static NotificationEvent ofSend(Long notificationId) { return NotificationEvent.builder() .notificationId(notificationId) .eventType("SEND") .timestamp(LocalDateTime.now()) .build(); } // Factory method: create a RETRY event public static NotificationEvent ofRetry(Long notificationId) { return NotificationEvent.builder() .notificationId(notificationId) .eventType("RETRY") .timestamp(LocalDateTime.now()) .build(); } }
Why only the ID — not the full notification?
Sending the full entity over Kafka is an anti-pattern. The DB is the source of truth. If the notification is updated between publish and consume (e.g., user cancels it), the consumer always sees the latest state by fetching from DB. It also keeps Kafka messages tiny (~100 bytes) — enabling higher throughput.
KAFKA MESSAGE FORMAT (JSON) Key: "42" ← String(notificationId) Value: { ← JSON serialized NotificationEvent "notificationId": 42, "eventType": "SEND", "timestamp": "2026-03-18T19:00:00" } Total message size: ~90 bytes Compare: Full Notification entity would be ~2KB (20x larger)

7 The Producer — NotificationKafkaProducer.java
com/notification/kafka/NotificationKafkaProducer.java
@Slf4j @Component @RequiredArgsConstructor public class NotificationKafkaProducer { private final KafkaTemplate<String, NotificationEvent> kafkaTemplate; @Value("${notification.kafka.topic.send:notification-send}") private String sendTopic; @Value("${notification.kafka.topic.retry:notification-retry}") private String retryTopic; // ── Publish to notification-send topic ── public void sendNotificationEvent(NotificationEvent event) { String key = String.valueOf(event.getNotificationId()); CompletableFuture<SendResult<String, NotificationEvent>> future = kafkaTemplate.send(sendTopic, key, event); future.whenComplete((result, ex) -> { if (ex != null) { log.error("Failed to publish [{}] to [{}]: {}", event.getNotificationId(), sendTopic, ex.getMessage()); } else { log.info("Published [{}] to [{}] partition={} offset={}", event.getNotificationId(), sendTopic, result.getRecordMetadata().partition(), result.getRecordMetadata().offset()); } }); } // ── Publish to notification-retry topic ── public void sendRetryEvent(NotificationEvent event) { String key = String.valueOf(event.getNotificationId()); kafkaTemplate.send(retryTopic, key, event) .whenComplete((result, ex) -> { if (ex != null) log.error("Failed to publish retry [{}]: {}", event.getNotificationId(), ex.getMessage()); else log.info("Published retry [{}] to [{}] p={} o={}", event.getNotificationId(), retryTopic, result.getRecordMetadata().partition(), result.getRecordMetadata().offset()); }); } }
Design Decisions:
• Async publish with callback — send() returns a CompletableFuture; HTTP thread is never blocked waiting for Kafka ACK.
• Notification ID as key — Kafka hashes the key to determine the partition. Same ID → same partition → ordering preserved for that notification.
• Callback logging — whenComplete() logs partition and offset for observability and debugging.
8 The Consumer — NotificationKafkaConsumer.java
com/notification/kafka/NotificationKafkaConsumer.java
@Slf4j @Component @RequiredArgsConstructor public class NotificationKafkaConsumer { private final NotificationRepository notificationRepository; private final NotificationProcessingService processingService; private final RetryService retryService; // ══════════════════ MAIN SEND LISTENER ══════════════════ @KafkaListener( topics = "${notification.kafka.topic.send:notification-send}", groupId = "${notification.kafka.consumer.group-id:notification-group}", containerFactory = "kafkaListenerContainerFactory" ) public void consumeSendEvent(NotificationEvent event) { log.info("Consumed SEND event for notification [{}]", event.getNotificationId()); Notification notification = notificationRepository .findById(event.getNotificationId()) .orElseThrow(() -> new RuntimeException( "Notification not found: " + event.getNotificationId())); processingService.processNotification(notification); } // ══════════════════ RETRY LISTENER ══════════════════ @KafkaListener( topics = "${notification.kafka.topic.retry:notification-retry}", groupId = "${notification.kafka.consumer.group-id:notification-group}", containerFactory = "kafkaListenerContainerFactory" ) public void consumeRetryEvent(NotificationEvent event) { log.info("Consumed RETRY event for notification [{}]", event.getNotificationId()); Notification notification = notificationRepository .findById(event.getNotificationId()) .orElseThrow(() -> new RuntimeException( "Notification not found: " + event.getNotificationId())); retryService.executeRetry(notification); } // ══════════════════ DEAD LETTER LISTENERS ══════════════════ @KafkaListener(topics = "notification-send.DLT", groupId = "notification-group-dlt") public void consumeSendDlt(NotificationEvent event) { log.error("DLT: Notification [{}] failed all Kafka retry attempts. " + "Manual intervention required.", event.getNotificationId()); } @KafkaListener(topics = "notification-retry.DLT", groupId = "notification-group-dlt") public void consumeRetryDlt(NotificationEvent event) { log.error("DLT: Retry for notification [{}] permanently failed.", event.getNotificationId()); } }
Consumer error handling flow:
1. Consumer calls processNotification() — if it throws...
2. DefaultErrorHandler catches the exception and retries 2 more times (1s apart)
3. After 3 total failures → DeadLetterPublishingRecoverer routes the message to .DLT topic
4. DLT consumer logs the error for manual investigation
5. Offset committed — consumer moves on to the next message (no blocking)
9 Service Layer Changes — NotificationService.java

The key change: replace the direct @Async method call with a Kafka event publish.

❌ Before — Direct @Async Call
// NotificationService.java (BEFORE) private final NotificationProcessingService processingService; notification = notificationRepository.save(notification); // Direct in-process async call — same JVM processingService.processNotificationAsync(notification); return NotificationResponse.fromEntity(notification);
✅ After — Kafka Event Publish
// NotificationService.java (AFTER) private final NotificationKafkaProducer kafkaProducer; notification = notificationRepository.save(notification); // Publish lightweight event to Kafka — fully decoupled kafkaProducer.sendNotificationEvent( NotificationEvent.ofSend(notification.getId())); return NotificationResponse.fromEntity(notification);
What exactly changed:
• processingService dependency removed from NotificationService
• kafkaProducer dependency added to NotificationService
• Both sendNotification() and sendBulkNotifications() now publish to Kafka
• The NotificationKafkaConsumer (on the other side of Kafka) calls processingService.processNotification()
• NotificationServiceTest updated: mocks kafkaProducer instead of processingService
10 Dead Letter Topics — Handling Permanent Failures

A Dead Letter Topic (DLT) is a special Kafka topic where messages that fail all processing attempts are automatically routed. Instead of losing the message or blocking the consumer, it is safely parked for later investigation.

DLT LIFECYCLE — FROM FAILURE TO RESOLUTION ┌──────────────┐ ┌──────────────────┐ ┌───────────────────┐ │ notification │ fail │ DefaultError │ fail │ DeadLetter │ │ -send topic │ ──────▶│ Handler │ ─────▶│ Publishing │ │ │ ×3 │ │ all │ Recoverer │ │ [message] │ │ Retry 1 (1s) │ │ │ │ │ │ Retry 2 (1s) │ │ Publishes to: │ └──────────────┘ │ Retry 3 (final) │ │ notification │ └──────────────────┘ │ -send.DLT │ └────────┬──────────┘ │ ▼ ┌───────────────────┐ │ DLT Consumer │ │ consumeSendDlt() │ │ │ │ Actions: │ │ • Log error │ │ • Alert ops team │ │ • Store for │ │ manual review │ └───────────────────┘ Resolution options: 1. Fix the root cause, then replay messages from DLT 2. Manually update notification status in DB 3. Forward to a different processing pipeline
How DLT is configured
Error handler: DefaultErrorHandler
Recovery strategy: DeadLetterPublishingRecoverer
Backoff: FixedBackOff(1000ms, 2 retries)
Total attempts: 3 (1 original + 2 retries)
DLT naming: {original-topic}.DLT (Spring Kafka convention)
When does a message go to DLT?
Database down: Cannot fetch notification by ID
Channel handler error: Email server unreachable × 3
Serialization error: Corrupted message format
Unknown exception: Any unhandled RuntimeException
NOT sent to DLT: Successful processing (obviously)

12 Complete Comparison — @Async vs Kafka
Capability@Async Thread PoolKafka Message Queue
Message persistence In-memory only — lost on crash Disk-based — survives broker restarts
Horizontal scaling Same JVM only Add instances → Kafka auto-rebalances
Replay capability No — task consumed once Yes — reset offset to replay from any point
Backpressure CallerRunsPolicy blocks HTTP thread Consumers pull at their own pace
Dead letter handling Manual try/catch only Built-in DLT with auto-routing
Monitoring JMX thread pool metrics only Consumer lag, partition offsets, group metrics
Message ordering No guarantee — thread pool is unordered Per-partition ordering by key
Decoupling Producer + consumer in same JVM Fully independent processes
Infrastructure cost None — built into Spring Requires Kafka broker (Docker/cloud)
Complexity Simple — just @Async annotation Moderate — topics, serializers, config
Latency ~0ms (same JVM call) ~1-5ms (network hop to broker)
Fault tolerance App crash = data loss Broker crash = automatic failover with replicas

15 How to Run Kafka Locally
1
Start Kafka using Docker (simplest approach)
docker run -d --name kafka -p 9092:9092 apache/kafka:3.6.1
2
Start the Spring Boot application
mvn spring-boot:run
Topics are auto-created by the NewTopic beans in KafkaConfig.java.
3
Send a test notification
POST http://localhost:8080/api/notifications/send with your JSON payload.
4
Monitor Kafka topics (optional)
docker exec kafka /opt/kafka/bin/kafka-console-consumer.sh --bootstrap-server localhost:9092 --topic notification-send --from-beginning
