📨 Why Kafka? The Problem with Direct @Async

Our previous architecture used Spring's @Async with a ThreadPoolTaskExecutor to process notifications in background threads. While this was a huge improvement over synchronous processing, it has fundamental limitations at scale:

❌ Problems with @Async Only
- In-memory queue — if app crashes, queued tasks are lost
- No replay — failed events cannot be re-consumed
- Single instance — can't distribute work across servers
- Tight coupling — producer and consumer in same JVM
- No backpressure — queue capacity is finite, CallerRunsPolicy blocks HTTP thread
- No ordering guarantees — thread pool doesn't guarantee order
✅ What Kafka Adds
- Persistent queue — messages survive broker restarts
- Replay capability — re-read messages from any offset
- Horizontal scaling — add consumer instances across servers
- Full decoupling — producer and consumer are independent
- Backpressure built-in — consumers pull at their own pace
- Partition ordering — same key always goes to same partition

🏗 Architecture Overview: Topics, Producers & Consumers

The Kafka integration introduces 4 topics, 1 producer, and 1 consumer component with 4 listener methods.

Kafka Topics

notification-send 3 partitions
Purpose: Main topic for new notifications that need to be processed and delivered.
Key: notificationId — ensures same notification always goes to same partition.
Producer: NotificationService publishes here after saving notification to DB.
Consumer: NotificationKafkaConsumer.consumeSendEvent() picks up and calls processNotification().
notification-retry 3 partitions
Purpose: Topic for notifications that failed and need to be retried.
Key: notificationId
Producer: RetryService can publish here for event-driven retries.
Consumer: NotificationKafkaConsumer.consumeRetryEvent() picks up and calls retryService.executeRetry().
notification-send.DLT Dead Letter
Purpose: Messages that failed 3 times on the notification-send topic land here.
Consumer: consumeSendDlt() logs the failure for manual intervention.
Action: Operations team investigates and replays or resolves manually.
notification-retry.DLT Dead Letter
Purpose: Messages that failed 3 times on the notification-retry topic land here.
Consumer: consumeRetryDlt() logs the failure for manual intervention.
Action: Permanent failures that need human review.

🔀 Message Flow: Step by Step

Flow 1: Sending a New Notification

1
HTTP Request arrives at /api/notifications/send
The controller calls NotificationService.sendNotification().
2
Save notification to database with status=PENDING
notificationRepository.save(notification) — notification gets a DB-generated ID.
3
Publish event to Kafka topic notification-send
kafkaProducer.sendNotificationEvent(NotificationEvent.ofSend(id))
The message key is the notificationId (ensures ordering per notification).
4
HTTP response returned immediately to client
Client gets NotificationResponse with status=PENDING. Processing happens asynchronously via Kafka.
5
Kafka consumer picks up the event
consumeSendEvent() fetches the notification from DB by ID, calls processNotification().
6
Notification is processed and delivered
The channel handler (Email/SMS/Push) sends the notification. Status updated to SENT or FAILED.

📦 Step 1: The Kafka Event DTO — NotificationEvent

The event carries minimal data: just the notificationId and eventType. The full notification data stays in the database — the consumer fetches it by ID. This is the event-carried state transfer pattern: lightweight events, heavy data stays in the source of truth (DB).

NotificationEvent.java
package com.notification.event; @Getter @Setter @NoArgsConstructor @AllArgsConstructor @Builder @ToString public class NotificationEvent implements Serializable { private Long notificationId; // DB primary key — consumer fetches full data private String eventType; // "SEND" or "RETRY" private LocalDateTime timestamp; // When the event was created // Factory methods for clean creation public static NotificationEvent ofSend(Long notificationId) { return NotificationEvent.builder() .notificationId(notificationId) .eventType("SEND") .timestamp(LocalDateTime.now()) .build(); } public static NotificationEvent ofRetry(Long notificationId) { /* same pattern */ } }
Why only the ID? Sending the full notification entity over Kafka is an anti-pattern. The DB is the source of truth. If the notification is updated between publish and consume (e.g., user cancels it), the consumer always sees the latest state by fetching from DB.
⚙ Step 2: Kafka Configuration — KafkaConfig.java

The configuration class creates topics, configures the producer and consumer factories, and sets up the Dead Letter Topic error handler. Every setting is explained below.

Topic Creation

KafkaConfig.java — Topics
@Bean public NewTopic notificationSendTopic() { return TopicBuilder.name("notification-send") .partitions(3) // 3 partitions for parallel consumption .replicas(1) // 1 replica (increase in production) .build(); } @Bean public NewTopic notificationRetryTopic() { return TopicBuilder.name("notification-retry") .partitions(3) .replicas(1) .build(); } // Dead Letter Topics (1 partition each — low volume) @Bean public NewTopic notificationSendDltTopic() { return TopicBuilder.name("notification-send.DLT").partitions(1).replicas(1).build(); }
Why 3 partitions? Each partition can be consumed by one consumer thread. With concurrency=3, we have 3 consumer threads, each assigned to 1 partition. This gives us 3x parallel processing. Increase partitions to scale further.

Producer Configuration

KafkaConfig.java — Producer
@Bean public ProducerFactory<String, NotificationEvent> producerFactory() { Map<String, Object> props = new HashMap<>(); props.put(BOOTSTRAP_SERVERS_CONFIG, "localhost:9092"); props.put(KEY_SERIALIZER_CLASS_CONFIG, StringSerializer.class); props.put(VALUE_SERIALIZER_CLASS_CONFIG, JsonSerializer.class); // Serialize event as JSON props.put(ACKS_CONFIG, "all"); // Wait for all replicas to ACK props.put(RETRIES_CONFIG, 3); // Retry on transient errors props.put(ENABLE_IDEMPOTENCE_CONFIG, true); // Prevent duplicate messages return new DefaultKafkaProducerFactory<>(props); }

📤 Step 3: The Producer — NotificationKafkaProducer
NotificationKafkaProducer.java
@Slf4j @Component @RequiredArgsConstructor public class NotificationKafkaProducer { private final KafkaTemplate<String, NotificationEvent> kafkaTemplate; @Value("${notification.kafka.topic.send:notification-send}") private String sendTopic; public void sendNotificationEvent(NotificationEvent event) { String key = String.valueOf(event.getNotificationId()); CompletableFuture<SendResult<...>> future = kafkaTemplate.send(sendTopic, key, event); future.whenComplete((result, ex) -> { if (ex != null) { log.error("Failed to publish: {}", ex.getMessage()); } else { log.info("Published [{}] to partition={}, offset={}", event.getNotificationId(), result.getRecordMetadata().partition(), result.getRecordMetadata().offset()); } }); } }
Key design decisions:
- Async publish with callback — send() returns a CompletableFuture; we don't block the HTTP thread waiting for Kafka ACK.
- Notification ID as key — Kafka hashes the key to determine the partition. Same notification ID always goes to the same partition, preserving order for that notification.
- Callback logging — whenComplete() logs the partition and offset for observability.
📥 Step 4: The Consumer — NotificationKafkaConsumer
NotificationKafkaConsumer.java
@Slf4j @Component @RequiredArgsConstructor public class NotificationKafkaConsumer { private final NotificationRepository notificationRepository; private final NotificationProcessingService processingService; private final RetryService retryService; // ==================== MAIN SEND TOPIC ==================== @KafkaListener( topics = "${notification.kafka.topic.send:notification-send}", groupId = "${notification.kafka.consumer.group-id:notification-group}", containerFactory = "kafkaListenerContainerFactory" ) public void consumeSendEvent(NotificationEvent event) { log.info("Consumed SEND event for notification [{}]", event.getNotificationId()); Notification notification = notificationRepository .findById(event.getNotificationId()) .orElseThrow(() -> new RuntimeException("Not found: " + event.getNotificationId())); processingService.processNotification(notification); } // ==================== RETRY TOPIC ==================== @KafkaListener(topics = "${notification.kafka.topic.retry:notification-retry}", ...) public void consumeRetryEvent(NotificationEvent event) { Notification notification = notificationRepository.findById(...).orElseThrow(...); retryService.executeRetry(notification); } // ==================== DEAD LETTER TOPICS ==================== @KafkaListener(topics = "notification-send.DLT", groupId = "notification-group-dlt") public void consumeSendDlt(NotificationEvent event) { log.error("DLT: Notification [{}] failed all Kafka attempts. Manual intervention required.", event.getNotificationId()); } @KafkaListener(topics = "notification-retry.DLT", groupId = "notification-group-dlt") public void consumeRetryDlt(NotificationEvent event) { log.error("DLT: Retry for notification [{}] failed all attempts.", event.getNotificationId()); } }
Consumer error handling flow:
1. Consumer calls processNotification() — if it throws an exception...
2. DefaultErrorHandler catches it and retries 2 more times (1 second apart)
3. If all 3 attempts fail, DeadLetterPublishingRecoverer publishes the message to .DLT topic
4. The DLT consumer logs the error for manual investigation
5. The offset is committed — the consumer moves on to the next message
🔄 Step 5: Service Layer Changes

NotificationService.java — Publish to Kafka

Before (Direct @Async)
notification = notificationRepository.save(notification); // Direct in-process async call processingService.processNotificationAsync(notification); return NotificationResponse.fromEntity(notification);
After (Kafka Publish)
notification = notificationRepository.save(notification); // Publish event to Kafka — fully decoupled kafkaProducer.sendNotificationEvent( NotificationEvent.ofSend(notification.getId())); return NotificationResponse.fromEntity(notification);
What changed: Instead of directly calling processingService.processNotificationAsync() (which runs in the same JVM's thread pool), we publish a lightweight event to Kafka. The consumer — potentially on a completely different server — picks up the event and processes it. The processingService dependency is removed from NotificationService.

📈 Scaling with Kafka

Kafka's consumer group protocol makes horizontal scaling simple:

Single Instance (current setup) ┌────────────────────────────┐ │ Instance 1 │ │ Consumer Group: notif-grp │ │ C-0 ← Partition 0 │ │ C-1 ← Partition 1 │ │ C-2 ← Partition 2 │ └────────────────────────────┘ Throughput: ~300 notifications/sec Two Instances (scale out — just start another app) ┌────────────────────────────┐ ┌────────────────────────────┐ │ Instance 1 │ │ Instance 2 │ │ Consumer Group: notif-grp │ │ Consumer Group: notif-grp │ │ C-0 ← Partition 0 │ │ C-0 ← Partition 2 │ │ C-1 ← Partition 1 │ │ │ └────────────────────────────┘ └────────────────────────────┘ Kafka automatically rebalances partitions across instances! Throughput: ~500 notifications/sec Three Instances (maximum parallelism for 3 partitions) ┌──────────────┐ ┌──────────────┐ ┌──────────────┐ │ Instance 1 │ │ Instance 2 │ │ Instance 3 │ │ C-0 ← P0 │ │ C-0 ← P1 │ │ C-0 ← P2 │ └──────────────┘ └──────────────┘ └──────────────┘ Throughput: ~900 notifications/sec Note: More instances than partitions = idle consumers. To scale beyond 3, increase partition count first.

🚀 How to Run Kafka Locally
1
Download and start Apache Kafka
Download from https://kafka.apache.org/downloads
Or use Docker: docker run -p 9092:9092 apache/kafka:3.6.1
2
Start Zookeeper (if not using KRaft mode)
bin/zookeeper-server-start.sh config/zookeeper.properties
3
Start Kafka Broker
bin/kafka-server-start.sh config/server.properties
4
Start the Spring Boot application
mvn spring-boot:run
Topics are auto-created by Spring Kafka's NewTopic beans.
5
Monitor topics with console consumer
bin/kafka-console-consumer.sh --bootstrap-server localhost:9092 --topic notification-send --from-beginning
Docker Compose (recommended for local dev): The upcoming Docker step (#5 in our roadmap) will include a docker-compose.yml with Kafka, Zookeeper, and the application pre-configured.
