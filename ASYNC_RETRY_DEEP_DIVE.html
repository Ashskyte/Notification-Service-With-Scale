Async Retry Deep Dive: Replacing Thread.sleep

Why blocking retries kill throughput, and how scheduled retries fix it • March 2026
🚨 The Problem: Thread.sleep Blocks the Async Thread Pool

In the previous implementation, when a notification failed to send, the RetryService used Thread.sleep(delay) to wait before retrying. This seems harmless but is catastrophic at scale.

RetryService.java (BEFORE — the problem)
public void retryNotification(Notification notification) { long delay = calculateExponentialBackoff(notification.getRetryCount()); try { Thread.sleep(delay); // ← BLOCKS the thread for 1-4+ seconds! handler.send(notification); // Try to send again notification.setStatus(SENT); } catch (Exception e) { notification.setStatus(RETRY); // Will retry again later } notificationRepository.save(notification); }
Why this is dangerous: This method runs on the notif-async-* thread pool (10 core threads). When Thread.sleep(1000) executes, that thread is completely blocked — doing nothing, just sleeping — but it's still counted as "in use" by the pool. No other notification can use that thread until the sleep finishes.
💥 What Happens at Scale: The Cascade Failure

Consider a realistic failure scenario: an email provider goes down for 30 seconds. During that window, every notification fails and enters the retry path.

Timeline: Email provider goes down at T=0 T=0s 100 notifications sent → 100 fail → 100 enter retryNotification() 10 threads call Thread.sleep(1000) — ALL 10 CORE THREADS BLOCKED Remaining 90 notifications queued, waiting for a free thread T=1s 10 threads wake up → try to send → fail again → Thread.sleep(2000) ALL 10 THREADS BLOCKED FOR 2 MORE SECONDS Meanwhile: new incoming notifications CANNOT be processed Queue grows: 90 + new arrivals = 200+ waiting T=3s 10 threads wake up → try to send → fail again → Thread.sleep(4000) ALL 10 THREADS BLOCKED FOR 4 MORE SECONDS Queue: 300+ notifications backed up NEW notifications arriving at /api/notifications/send are piling up T=7s 10 threads wake up → max retries exhausted → mark FAILED Finally start processing queued notifications — 7+ seconds late TOTAL DAMAGE: For 7+ seconds, the ENTIRE async thread pool was hijacked by sleeping retry threads. ZERO new notifications could be processed.
❌ With Thread.sleep
- Thread blocked for 1s, 2s, 4s per retry
- 10 failures = all 10 core threads sleeping
- New notifications can't be processed
- Exponential backoff = exponential blocking
- 7+ seconds of zero throughput
- System appears "hung" to users
✅ With Scheduled Retry
- Failure handler runs in <1ms (just DB save)
- Thread immediately returns to pool
- New notifications processed without delay
- Scheduler picks up retries when they're due
- Zero impact on throughput
- System stays responsive under failures
Metric Thread.sleep (Before) Scheduled Retry (After)
Thread blocked per failure 1,000 - 4,000+ ms < 1 ms (just a DB save)
10 concurrent failures ALL 10 core threads blocked 0 threads blocked

✅ The Solution: Database-Backed Scheduled Retry

Instead of blocking a thread with Thread.sleep, we:

1
On failure: Calculate the next retry time and save it to the database
Set status = RETRY and nextRetryAt = now + exponentialBackoff. The thread is released back to the pool immediately (<1ms).
2
Scheduler polls every 10 seconds for due retries
A @Scheduled job queries: SELECT * FROM notifications WHERE status = 'RETRY' AND nextRetryAt <= NOW(). Only processes notifications whose wait time has actually elapsed.
3
Execute the retry — success or schedule another
If the send succeeds → mark SENT. If it fails again and retries remain → calculate new nextRetryAt with increased backoff. If max retries exhausted → mark FAILED permanently.
New Flow: Failure → Schedule → Poll → Retry ┌────────────────────────────────────────────────────────────┐ │ processNotification() on async thread │ │ │ │ handler.send(notification) → THROWS EXCEPTION │ │ │ │ │ ▼ │ │ scheduleRetry(notification) │ │ ├── retryCount < maxRetries? │ │ │ YES → status = RETRY │ │ │ nextRetryAt = now + backoff(retryCount) │ │ │ save to DB → return immediately (<1ms) │ │ │ NO → status = FAILED (permanent) │ │ │ save to DB → return immediately │ │ │ │ │ Thread returned to pool — ready for new work │ └────────────────────────────────────────────────────────────┘ ... time passes (1s, 2s, 4s depending on backoff) ... ┌────────────────────────────────────────────────────────────┐ │ @Scheduled(fixedRate = 10000) ← Every 10 seconds │ │ retryFailedNotifications() │ │ │ │ │ ▼ │ │ SELECT * FROM notifications │ │ WHERE status = 'RETRY' │ │ AND nextRetryAt <= NOW() │ │ AND retryCount < maxRetries │ │ │ │ │ ▼ │ │ For each due notification: │ │ executeRetry(notification) │ │ ├── send succeeds → status = SENT ✓ │ │ └── send fails │ │ ├── retries left → new nextRetryAt, status=RETRY │ │ └── max reached → status = FAILED (permanent) │ └────────────────────────────────────────────────────────────┘
📄 Code Changes: Step by Step

1. New Field: Notification.java

Added nextRetryAt to track when each notification should be retried. This is the "alarm clock" — the scheduler only picks up rows where this time has passed.

Notification.java
private String failureReason; private LocalDateTime nextRetryAt; // ← NEW: when to retry next

Also added a database index for efficient scheduler queries:

Notification.java — @Table indexes
@Table(name = "notifications", indexes = { @Index(name = "idx_notification_status", columnList = "status"), @Index(name = "idx_notification_priority", columnList = "priority"), @Index(name = "idx_notification_scheduled_at", columnList = "scheduledAt"), @Index(name = "idx_notification_next_retry_at", columnList = "nextRetryAt") // ← NEW })

2. New Query: NotificationRepository.java

Added a query to find notifications whose retry time has passed.

NotificationRepository.java
@Query("SELECT n FROM Notification n " + "WHERE n.status = 'RETRY' " + "AND n.retryCount < n.maxRetries " + "AND n.nextRetryAt <= :now " + "ORDER BY n.nextRetryAt ASC") List<Notification> findDueRetryNotifications(@Param("now") LocalDateTime now);

3. Rewritten: RetryService.java

The old retryNotification() with Thread.sleep is replaced by three clean methods:

RetryService.java — scheduleRetry()
/** * Called immediately when a notification fails. * Sets status=RETRY and nextRetryAt, then RETURNS IMMEDIATELY. * No Thread.sleep — the thread is free in <1ms. */ @Transactional public void scheduleRetry(Notification notification) { if (notification.getRetryCount() >= notification.getMaxRetries()) { notification.setStatus(NotificationStatus.FAILED); notificationRepository.save(notification); return; } long delayMs = calculateExponentialBackoff(notification.getRetryCount()); LocalDateTime nextRetryAt = LocalDateTime.now().plusNanos(delayMs * 1_000_000); notification.setStatus(NotificationStatus.RETRY); notification.setNextRetryAt(nextRetryAt); notificationRepository.save(notification); // Persisted to DB — survives restart }
RetryService.java — processDueRetries()
/** * Called by the scheduler every 10 seconds. * Finds RETRY notifications where nextRetryAt has passed. */ @Transactional public void processDueRetries() { List<Notification> dueRetries = notificationRepository.findDueRetryNotifications(LocalDateTime.now()); for (Notification notification : dueRetries) { executeRetry(notification); } }
RetryService.java — executeRetry()
/** * Attempts to send a single due-for-retry notification. * On success: SENT. On failure: schedule next retry or mark permanently FAILED. */ @Transactional public void executeRetry(Notification notification) { notification.setRetryCount(notification.getRetryCount() + 1); try { NotificationChannelHandler handler = channelResolverService.resolve(notification.getChannel()); handler.send(notification); notification.setStatus(NotificationStatus.SENT); notification.setSentAt(LocalDateTime.now()); notification.setNextRetryAt(null); // Clear — no more retries needed } catch (Exception e) { notification.setFailureReason(e.getMessage()); if (notification.getRetryCount() >= notification.getMaxRetries()) { notification.setStatus(NotificationStatus.FAILED); notification.setNextRetryAt(null); // Permanent failure } else { long delayMs = calculateExponentialBackoff(notification.getRetryCount()); notification.setNextRetryAt( LocalDateTime.now().plusNanos(delayMs * 1_000_000)); notification.setStatus(NotificationStatus.RETRY); } } notificationRepository.save(notification); }

⏱ Exponential Backoff: The Retry Timing

Both the old and new implementations use the same exponential backoff formula. The difference is how the delay is implemented — sleeping vs scheduling.

Exponential Backoff Formula
delay = initialDelayMs × multiplier ^ retryCount // With default config: initialDelayMs=1000, multiplier=2.0 Retry 1: 1000 × 2^0 = 1,000ms (1 second) Retry 2: 1000 × 2^1 = 2,000ms (2 seconds) Retry 3: 1000 × 2^2 = 4,000ms (4 seconds)
Retry Attempt Backoff Delay Thread.sleep (Before) Scheduled Retry (After)
1st 1,000ms Thread blocked 1s nextRetryAt = now + 1s
2nd 2,000ms Thread blocked 2s nextRetryAt = now + 2s
3rd (max) 4,000ms Thread blocked 4s nextRetryAt = now + 4s
Total thread blocked per notification 7+ seconds 0 seconds
