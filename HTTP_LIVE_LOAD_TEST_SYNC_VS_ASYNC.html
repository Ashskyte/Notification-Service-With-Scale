🧪 How HttpLoadTest.java Works

HttpLoadTest.java is a standalone Java program (no Spring, no Maven needed to run). It uses Java 21's built-in java.net.http.HttpClient to send real HTTP requests to a running server. The file lives at src/test/java/com/notification/performance/HttpLoadTest.java in both repos.

1
Seed 10 test users via HTTP
Sends 10 sequential POST /api/users requests, each with a unique username (appends System.nanoTime() to avoid duplicates). Extracts the id from each JSON response and stores it in a list. These IDs are used for notification requests.
2
Warmup: 5 sequential requests
Sends 5 notification requests one at a time to prime the JVM's JIT compiler, Tomcat's thread pool, HikariCP connection pool, and the HTTP client's connection cache. Waits 500ms after warmup. This ensures the timed test measures steady-state performance, not cold-start overhead.
3
Create ExecutorService with 30 threads
A Executors.newFixedThreadPool(30) simulates 30 concurrent HTTP clients. A CountDownLatch(200) tracks completion. AtomicInteger counters track successes and failures in a thread-safe manner.
4
Fire 200 concurrent POST requests
Each of the 200 tasks is submitted to the executor. Each task:
  • Picks a user ID (round-robin: userIds.get(index % userIds.size()))
  • Builds a JSON body with userId, title, message, channel=EMAIL, priority=MEDIUM
  • Records System.currentTimeMillis() before and after the HTTP call
  • Stores the elapsed time in a synchronized latency list
  • Checks the HTTP status code (200/201 = success, anything else = failure)
  • Decrements the CountDownLatch
5
Await completion & compute metrics
latch.await(5, TimeUnit.MINUTES) blocks until all 200 tasks finish. Then the latency list is sorted and metrics are computed:
  • Total time = overallEnd - overallStart
  • Throughput = successCount * 1000 / totalTimeMs (requests/sec)
  • Avg latency = mean of all individual request durations
  • P50/P95/P99 = percentile from sorted latency list
6
Print formatted results
All metrics are printed to stdout in a structured format, making it easy to compare SYNC vs ASYNC runs side by side.
📖 Step-by-Step: Run It Yourself

Follow these exact steps to reproduce the benchmark on your machine. You need Java 21 and Maven installed.

Step 1: Clone Both Repositories
# Clone the SYNC repo (original blocking version) git clone https://github.com/Ashskyte/Notification-Service-Sync.git Notification-System # Clone the ASYNC repo (@Async + ThreadPool version) git clone https://github.com/Ashskyte/Notification-Service-With-Scale.git Notification-System-With-Scale
Step 2: Build & Start the SYNC Server
# Navigate to sync repo cd Notification-System # Build the project (skip tests for speed) mvn clean install -DskipTests # Start the server on port 8080 mvn spring-boot:run # Wait for: "Started NotificationSystemApplication in X seconds" # The server is now running at http://localhost:8080
Step 3: Run the HTTP Load Test Against SYNC Server
# Open a NEW terminal (keep the server running in the first one) cd Notification-System # Run the load test as a standalone Java file java src/test/java/com/notification/performance/HttpLoadTest.java "SYNC" # Expected output: # ========================================================== # HTTP LOAD TEST RESULTS: SYNC # ========================================================== # Total requests : 200 # Successful (2xx) : 200 # Throughput : ~265 requests/sec # Avg latency : ~105 ms # ==========================================================
Note: The "SYNC" argument is just a label printed in the output. You can pass any string (or nothing) — it doesn't change test behavior.
Step 4: Stop the SYNC Server
# Go to the terminal running the sync server # Press Ctrl+C to stop it # Verify port 8080 is free (Windows) netstat -ano | findstr :8080 # If still occupied, force kill (replace PID with actual PID) taskkill /PID <PID> /F
Step 5: Build & Start the ASYNC Server
# Navigate to async repo cd Notification-System-With-Scale # Build the project mvn clean install -DskipTests # Start the server on port 8080 mvn spring-boot:run # Wait for: "Notification async executor initialized: core=10, max=50, queue=10000" # This confirms the async thread pool is active
Step 6: Run the HTTP Load Test Against ASYNC Server
# Open a NEW terminal cd Notification-System-With-Scale # Run the same load test java src/test/java/com/notification/performance/HttpLoadTest.java "ASYNC" # Expected output: # ========================================================== # HTTP LOAD TEST RESULTS: ASYNC # ========================================================== # Total requests : 200 # Successful (2xx) : 200 # Throughput : ~376 requests/sec # Avg latency : ~75 ms # ==========================================================
Step 7: Compare the Results

Place the two outputs side by side. The async server should show: lower total time, higher throughput, lower avg/median latency, and lower P95/P99 percentiles. The improvement factor will vary by machine but should be approximately 1.3x–1.7x across all HTTP metrics.

🔍 What Each HTTP Request Does on the Server
Sync Server (Blocking)
1. Tomcat thread receives request
2. Validate JSON → deserialize to SendNotificationRequest
3. Look up user from DB (~3ms)
4. Save notification to DB with status=PENDING (~5ms)
5. BLOCKS: Send email via channel handler (100ms)
6. Update notification status to SENT (~3ms)
7. Serialize response JSON → return HTTP 201

Total: ~111ms+ per request
Tomcat thread is blocked for the entire duration
Async Server (Non-Blocking)
1. Tomcat thread receives request
2. Validate JSON → deserialize to SendNotificationRequest
3. Look up user from DB (~3ms)
4. Save notification to DB with status=PENDING (~5ms)
5. ASYNC: Submit to notificationTaskExecutor queue (~0.01ms)
6. Serialize response JSON → return HTTP 201

Total: ~8ms per request
Email delivery happens in background thread pool (10-50 threads)
🧠 Why 1.4x (HTTP) vs 3.6x (JUnit)?

The JUnit test measured 3.6x improvement because it called notificationService.sendNotification() directly — no HTTP overhead.


The HTTP test adds fixed costs that both versions pay equally:

~1-3ms Network round-trip (localhost loopback) ~5-10ms Tomcat thread scheduling & connection handling ~2-5ms JSON serialization / deserialization ~1-2ms Spring filter chain (validation, error handling) ~5-15ms HTTP connection pool contention (30 concurrent clients)

These ~15-35ms of fixed overhead compress the relative ratio. But the absolute savings remain significant:


P50 dropped from 120ms to 71ms — a 49ms saving per request. At 1,000 requests/sec, that's 49 seconds of user-facing wait time saved every second. At production scale with real external APIs (200-2000ms instead of 100ms), the async advantage grows to 5-20x.
