📐 Architecture Design Decisions

⚖️ Load Balancer

  • L7 Load Balancer (Nginx / AWS ALB)
  • Health checks every 10s on /actuator/health
  • SSL/TLS termination at LB layer
  • Active-Active deployment for HA
  • Auto-scaling group behind LB

🛡️ Rate Limiting Strategy

  • Token Bucket algorithm (Redis-backed)
  • Per-user: 100 requests/minute
  • Per-IP: 1000 requests/minute
  • Bulk API: 10 requests/minute
  • HTTP 429 Too Many Requests on breach

🚪 API Gateway

  • JWT / OAuth2 authentication
  • Request validation & sanitization
  • API versioning (v1, v2)
  • Request logging & correlation ID injection
  • Circuit breaker for downstream protection

📨 Kafka Integration

  • 3+ broker cluster, Replication Factor = 3
  • Topics: notification-send, notification-retry
  • 12+ partitions for parallel consumer processing
  • Consumer group: notification-group (concurrency=3+)
  • Dead Letter Topics for failed messages
  • Key = notificationId for partition ordering

🗄️ Database Design

  • PostgreSQL with Primary-Replica replication
  • Tables: users, notifications, user_channel_preferences
  • Indexed: user_id, status, priority, scheduled_at
  • Connection pool: HikariCP (max 30 connections)
  • Read replicas for query offloading
  • Future: Table partitioning by date for archival

📦 Scalability Strategy

  • Stateless service → horizontal scaling
  • Kubernetes Deployment with HPA
  • Auto-scale: 2 (min) → 10 (max) pods
  • Async thread pool: core=10, max=50, queue=10K
  • Kafka consumers scale with partitions
  • Batch processing for bulk notifications (batch=100)

🔄 Retry & Failure Handling

  • Exponential backoff: 1s → 2s → 4s
  • Max 3 retry attempts per notification
  • Retry events published to notification-retry topic
  • Exhausted retries → Dead Letter Topic
  • Status tracking: PENDING → PROCESSING → SENT / FAILED / RETRY

🔌 High Availability

  • Multi-AZ deployment (cloud)
  • LB with active health checks
  • Kafka cluster: no single point of failure
  • DB: Primary-Replica with auto-failover
  • Redis Cluster with sentinel/HA mode
  • Graceful degradation on partial failures

🔀 Request Flow — End to End

1️⃣ Client sends POST /api/notifications/send
→
2️⃣ Load Balancer routes to healthy instance
→
3️⃣ API Gateway authenticates & rate-limits
→
4️⃣ Notification Service validates & persists to DB
→
5️⃣ Event published to Kafka (notification-send)
→
6️⃣ Worker consumes event, resolves channel
→
7️⃣ Channel Handler delivers (Email/SMS/Push)
→
8️⃣ Status updated to SENT ✅ (or RETRY on failure)
